НАЙМЕНУВАННЯ ТА ГАЛУЗЬ ЗАСТОСУВАННЯ
Назва розробки: Веб застосування для генерації тестових питань за теоретичними даними. 
Галузь застосування: 
Наведене технічне завдання поширюється на розробку веб забезпечення для генерації тестових питань за теоретичними даними 045440, котра призначена для оптимізації процесу підготовки до контрольних робіт та тестувань шляхом автоматизації процесу генерації тестових питань.
ПІДСТАВА ДЛЯ РОЗРОБКИ
Підставою для розробки веб застосування для генерації тестових питань за теоретичними даними є індивідуальний навчальний план студента та РНП затверджена кафедрою інформатики та програмної інженерії Національного технічного університету України «Київський політехнічний інститут імені Ігоря Сікорського».
ПРИЗНАЧЕННЯ РОЗРОБКИ
Розробка призначена для генерації тестових питань на основі наданих теоретичних даних з метою оптимізації освітніх процесів. Цільовою аудиторією розробки є викладачі, репетитори та учні або студенти. 
Метою розробки є полегшення процесу підготовки вчителів, викладачів та репетиторів до занять та забезпечення ефективного контролю засвоєння учнями знань шляхом реалізації генерації тестових запитань на основі теоретичних даних та налаштованих параметрів.
ВИМОГИ ДО ПРОГРАМНОГО ЗАБЕЗПЕЧЕННЯ
Вимоги до функціональних характеристик
Програмне забезпечення повинно забезпечувати виконання наступних основних функції:
Користувацького інтерфейсу
Завантаження теоретичних матеріалів – користувач має змогу завантажити файл або надрукувати вручну теоретичні матеріали для підготовки тестових питань (рисунок 4.1). Користувач має змогу перейти до налаштування параметрів (рисунок 4.2).

Рисунок 4.1 – Завантаження теоретичних матеріалів
Налаштування параметрів – користувач має змогу налаштовувати параметри тестових запитань, обираючи кількість запитань відповідного типу та рівень складності. Користувач має змогу переглянути список попередньо виділених ключових слів і видалити деякі, за необхідності (рисунок 4.3). Також користувач має змогу почати генерацію за допомогою натискання на відповідну кнопку (рисунок 4.2).

Рисунок 4.2 – Налаштування параметрів генерації

Рисунок 4.3 – Показ ключових слів
Екран прогресу генерації – користувач має бачити прогрес генерації, обрані ним параметри та, за потреби, скасовувати її або ставити на паузу (рисунок 4.4).

Рисунок 4.4 – Процес генерації
Пауза генерації – користувач має можливість поставити генерацію на паузу і відновити або скасувати її (рисунок 4.5). 

Рисунок 4.5 – Екран паузи генерації
Перегляд згенерованих тестів – користувач має змогу переглянути результат генерації(рисунок 4.6). Якщо результати не задовольняють користувача, він має мати можливість здійснити повторну генерацію, натиснувши на відповідну кнопку. При повторній генерації користувач має змогу повторно налаштувати параметри (рисунок 4.2). Також користувач має змогу озвучити згенеровані запитання, натиснувши на відповідну кнопку.

Рисунок 4.6 – Перегляд результату генерації
Експорт результату – користувач має змогу експортувати результати у вибраний формат через меню експорту (рисунок 4.7).

	Рисунок 4.7 – Меню експорту
Для користувача:
завантаження теоретичних відомостей у вигляді простого тексту або файлу у форматі .pdf, .doc або .txt;
налаштування параметрів генерації: вибір кількості запитань кожного типу;
вибір складності згенерованих запитань;
перегляд ключових слів перед генерацією;
видалення ключових слів;
скасування або пауза генерації запитань;
відновлення процесу генерації запитань після паузи;
експорт згенерованих запитань;
вибір формату експорту;
озвучка згенерованих запитань;
повторна генерація.
Додаткові вимоги:
час генерації не має перевищувати 10 хвилин;
платформа має підтримувати одночасне обслуговування щонайменше 100 користувачів;
система має використовувати протокол HTTPS для безпечного з’єднання;
автоматичне завершення сесії, якщо користувач неактивний більше 30 хвилин;
адаптивний дизайн застосунку;
підтримка кросбраузерності;
обмеження обсягу теоретичних даних (500 – 1000000).
Вимоги до надійності
– 	платформа має відновлювати працездатність не пізніше ніж за 1 хвилину після збою із збереженим станом користувача;
	–	система має забезпечувати зберігання резервних  копій прогресу в межах однієї сесії для відновлення після збоїв та помилок у процесі відображення або експорту результатів.
Умови експлуатації
Умови експлуатації згідно СанПін 2.2.2.542 – 96.
Вимоги до складу і параметрів технічних засобів
Програмне забезпечення повинно функціонувати на персональних комп’ютерах, ноутбуках та мобільних пристроях з доступом до мережі Інтернет.
Мінімальна конфігурація клієнтських технічних засобів:
браузер: Google Chrome 90+, Mozilla Firefox 88+, Microsoft Edge 90+, Safari 15;
Інтернет-з’єднання: 80 Мбіт/с;
роздільна здатність: 1024x768 px.
Рекомендована конфігурація клієнтських технічних засобів:
браузер: Google Chrome 135, Mozilla Firefox 137, Microsoft Edge 132, Safari 18.3;
Інтернет-з’єднання: 150 Мбіт/с;
роздільна здатність: 1920x1080 (Full HD).
Мінімальна конфігурація серверних технічних засобів:
тип хостингу: хмарний сервер;
процесор: 1 ядро;
оперативна пам’ять: 512 МБ;
дисковий простір: 1 ГБ SSD;
наявність HTTPS.
Рекомендована конфігурація серверних технічних засобів:
тип хостингу: хмарний сервер;
процесор: 1 ядро;
оперативна пам’ять: 1 ГБ;
дисковий простір: 2 ГБ SSD;
наявність HTTPS.
Вимоги до інформаційної та програмної сумісності
Програмне забезпечення повинно працювати під управлінням сучасних операційних систем, сумісних із браузерами, що підтримують JavaScript. 
Вимоги до вхідних даних
Вхідні дані повинні бути представлені як файли в форматі .pdf, .doc або .txt або як простий текст українською мовою від 500 до 1000000 слів. Підтримується лише текстова інформація без вкладених зображень, мультимедіа, таблиць чи інших складних форматів.
Вимоги до вихідних даних
Результати повинні бути виведені на веб-сторінку для попереднього перегляду. Має бути можливість експортувати результати у форматі .pdf, .doc, .txt, .html.
Вимоги до мови розробки
Клієнтську частину (інтерфейс) розробки виконати на мові програмування JavaScript ES2023 із застосуванням фреймворку React v19.
Серверну частину розробки виконати на Node.js 23.11.0 з використанням Express.js 5.1.0.
Для реалізації NLP-алгоритмів використати мову програмування Python 3.12.0.
Вимоги до середовища розробки
Розробку виконати за допомогою середовища Visual Studio Code (2023).
Вимоги до представленню вихідних кодів 
Вихідний код програми має бути представлений у вигляді GitHub репозиторію з усіма необхідними файлами.
